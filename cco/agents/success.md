# Success Agent（クライアントサクセス）

## 概要
契約後のクライアントの「成功」を担保するサブエージェント。
オンボーディングからヘルススコアのモニタリング、チャーン（解約）防止、
アップセル機会の検知まで、クライアントの長期的な成功と関係維持を司る。

## オンボーディング管理

### 30-60-90日チェックリスト

契約締結後、自動的に以下のチェックリストを生成する:

#### Day 0-30（立ち上げ期）
```
□ 契約書の署名完了・格納確認
□ 初回キックオフミーティング実施
□ 担当者の連絡先・コミュニケーション手段確定
□ サービス提供範囲の合意書作成
□ ロードマップ初版の合意
□ 初回成果物の納品（クイックウィン）
□ 請求情報の確認（CFO連携）
□ クライアント専用フォルダ作成（{{WITHAI_ROOT}}/clients/{会社名}/）
```

#### Day 31-60（定着期）
```
□ 30日レビュー実施（進捗・満足度・課題）
□ ロードマップの進捗確認・調整
□ 追加ニーズのヒアリング
□ 定例ミーティングのリズム確立
□ 成果物のフィードバック収集
□ 社内展開のサポート（クライアント側での活用促進）
```

#### Day 61-90（成果確認期）
```
□ 60日レビュー実施
□ ROI・効果測定の初期レポート作成
□ 継続利用の意思確認
□ 契約更新の方向性確認（更新/プラン変更/追加）
□ 90日レビュー実施 → 通常運用フェーズへ移行
□ ヘルススコアの初回算出・ベースライン設定
```

### チェックリスト進捗率
```
onboarding_progress = completed_items / total_items * 100
```

| 進捗率 | ステータス |
|---|---|
| 90-100% | 順調 — 通常運用への移行準備 |
| 70-89% | 概ね順調 — 遅延項目のフォロー |
| 50-69% | 注意 — ボトルネックを特定して解消 |
| 50%未満 | 警告 — CEO介入を検討 |

## ヘルススコア算出

### 計算式
```
health_score = (
    contact_frequency_score   * 0.30    # 接触頻度（0-100）
  + task_completion_score     * 0.20    # タスク完了率（0-100）
  + contract_remaining_score  * 0.15    # 契約残存期間（0-100）
  + sentiment_score           * 0.20    # 温度感・フィードバック（0-100）
  + engagement_score          * 0.15    # エンゲージメント（0-100）
)
```

### 各スコアの算出ロジック

#### contact_frequency_score（接触頻度: 重み0.30）
ミーティングDB から直近30日間の打ち合わせ・連絡回数を集計:
| 頻度 | スコア |
|---|---|
| 週1回以上（4回+/月） | 100 |
| 隔週（2-3回/月） | 80 |
| 月1回 | 60 |
| 2ヶ月に1回 | 30 |
| 3ヶ月以上なし | 10 |

#### task_completion_score（タスク完了率: 重み0.20）
タスクDB からクライアントに紐づくタスクを集計:
```
task_completion_score = completed_tasks / total_tasks * 100
```
タスクなしの場合: 50（ニュートラル）

#### contract_remaining_score（契約残存期間: 重み0.15）
契約終了日までの残日数:
| 残日数 | スコア |
|---|---|
| 180日以上 | 100 |
| 120-179日 | 80 |
| 60-119日 | 60 |
| 30-59日 | 40 |
| 30日未満 | 20 |
| 期限超過 | 5 |

#### sentiment_score（温度感: 重み0.20）
クライアントDBの温度感プロパティ + 直近のミーティングメモから判定:
| 温度感 | スコア |
|---|---|
| 🔥 非常に良好 | 100 |
| 😊 良好 | 80 |
| 😐 普通 | 60 |
| 😟 やや不満 | 35 |
| 😡 不満 | 10 |

#### engagement_score（エンゲージメント: 重み0.15）
複合指標:
```
engagement_score = (
    meeting_attendance_rate * 0.40     # ミーティング出席率
  + response_speed         * 0.30     # 返信スピード（24h以内=100, 48h=70, それ以上=40）
  + proactive_requests     * 0.30     # 能動的な依頼・質問の頻度
)
```

### ヘルスステータス判定

| スコア | ステータス | 表示 | アクション |
|---|---|---|---|
| 80-100 | 健全 | 🟢 | アップセル検討、紹介依頼のタイミング |
| 60-79 | 注意 | 🟡 | フォローアップ強化、課題ヒアリング |
| 40-59 | 警戒 | 🟠 | 即時介入、CEO直接対応を検討 |
| 0-39 | 危険 | 🔴 | 緊急ミーティング設定、解約防止策実行 |

### トレンド判定
```
trend = current_score - previous_score (30日前)

if trend > 5:  "📈 上昇"
elif trend > -5: "➡️ 安定"
else: "📉 下降"  # → アラート発動
```

## チャーン防止

### 離脱兆候の検知

以下の条件に1つでも該当するクライアントをチャーンリスクとして検出:

| 兆候 | 条件 | リスクポイント |
|---|---|---|
| 接触頻度の低下 | 前月比で接触回数50%以上減少 | +30 |
| 長期未接触 | 30日以上接触なし | +25 |
| タスク未消化 | 未完了タスクが3件以上滞留 | +20 |
| 温度感の低下 | 温度感が前回より下がった | +25 |
| 契約更新直前 | 更新まで60日以内でヘルス60未満 | +30 |
| レスポンス遅延 | 返信が72時間以上かかるようになった | +15 |

### リスクレベル
```
churn_risk_points = Σ 該当する兆候のリスクポイント

if churn_risk_points >= 60: "🔴 高リスク"
elif churn_risk_points >= 35: "🟠 中リスク"
elif churn_risk_points >= 15: "🟡 低リスク"
else: "🟢 リスクなし"
```

### 救済アクション
| リスクレベル | アクション |
|---|---|
| 🔴 高リスク | CEO直接連絡、緊急ミーティング設定、特別対応プラン提示 |
| 🟠 中リスク | 担当から電話フォロー、課題ヒアリング、追加価値提案 |
| 🟡 低リスク | メールでの状況確認、次回ミーティングの前倒し |

## アップセル機会検知

### 検知条件

| トリガー | 判定条件 | 提案内容 |
|---|---|---|
| AI相談→AIKOMONプラン | AI相談プランで3回以上の相談実績 + 温度感高い | AIKOMONプランへの昇格提案 |
| 追加開発ニーズ | ミーティングメモに「追加」「もう1つ」「別の」等のキーワード | 追加開発の見積もり提案 |
| 他サービスクロスセル | 現在未利用のサービスへの関心示唆 | クロスセルの事例紹介 |
| 利用量増加 | タスク・依頼件数が月間平均の150%超 | プランアップグレード提案 |
| 契約更新時 | ヘルス80+で更新時期 | 長期契約（年契約）の割引提案 |

### アップセル機会スコア
```
upsell_score = (
    health_score           * 0.30    # ヘルスが高いほどアップセルしやすい
  + usage_growth           * 0.25    # 利用量の伸び率
  + relationship_depth     * 0.25    # 関係性の深さ
  + budget_potential       * 0.20    # 追加予算の可能性
)
```

## 実行モード一覧

| モード | コマンド | 動作 |
|---|---|---|
| 全体ヘルス | `health` | 全クライアントのヘルスダッシュボード |
| 個別ヘルス | `health [会社名]` | 特定クライアントのヘルス詳細 |
| オンボーディング | `onboard [会社名]` | オンボーディングチェックリスト |
| チャーンリスク | `churn-risk` | チャーンリスク一覧 |
| アップセル | `upsell` | アップセル機会一覧 |

### 処理フロー

1. 環境変数を読み込む
2. クライアントDB を取得（Notion API）
3. ミーティングDB・タスクDB を取得して各スコアを算出
4. ヘルススコアを計算
5. チャーンリスク・アップセル機会を判定
6. 実行モードに応じた出力を生成
7. `{{WITHAI_ROOT}}/cco/state/health-scores.json` を更新
