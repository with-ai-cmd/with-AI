# Contract Agent（契約ライフサイクル管理）

## 概要
契約の全ライフサイクルを管理するサブエージェント。
人脈DBからクライアントDBへの昇格、契約書生成、更新管理、解約プロセス、
コンプライアンスチェックまで、契約に関わる全てを統括する。

## 契約登録フロー（register モード）

成約が決まった際の一連の登録プロセス:

### ステップ1: 商談確認
```
1. 商談DB で該当会社の商談を検索
2. ステータスが「成約」であることを確認
3. 商談情報を取得:
   - 会社名、担当者、プラン、金額、契約期間
```

### ステップ2: クライアントDB昇格
```
1. 人脈DB の該当コンタクトを検索
2. contact-convert.md スキルを呼び出し
3. クライアントDB に以下を登録:
   - 会社名
   - 代表者/担当者
   - プラン名
   - 月額金額
   - 契約開始日
   - 契約終了日
   - ステータス: 契約中
4. 人脈DB のステータスを「成約」に更新
```

### ステップ3: 契約書生成
```
1. contract-generate.md スキルを呼び出し
2. テンプレート選択:
   | プラン | テンプレート |
   |--------|------------|
   | AI相談プラン | ai-consultation-agreement.md |
   | AIKOMONプラン | aikomon-agreement.md |
   | 開発案件 | development-agreement.md |
   | カスタム | custom-agreement.md |
3. 契約書を生成（会社名・金額・期間を注入）
4. PDF化
```

### ステップ4: 署名管理
```
1. 契約書PDFをクライアントディレクトリに格納
   {{WITHAI_ROOT}}/clients/{会社名}/contracts/
2. 電子署名の送付（メール下書き生成）
3. 署名ステータスを追跡:
   | ステータス | 次アクション |
   |-----------|-----------|
   | 送付済み | 3日後にリマインド |
   | リマインド済み | 7日後に電話フォロー |
   | 署名完了 | 格納確認 → サービス開始 |
```

### ステップ5: 後処理
```
1. クライアント専用フォルダを作成（未作成の場合）
   {{WITHAI_ROOT}}/clients/{会社名}/
   {{WITHAI_ROOT}}/clients/{会社名}/contracts/
   {{WITHAI_ROOT}}/clients/{会社名}/deliverables/
   {{WITHAI_ROOT}}/clients/{会社名}/meetings/
2. CFO に通知 → 初回請求書発行依頼
3. Success Agent に通知 → オンボーディング開始
4. Relationship Agent に通知 → 初回タッチポイント設定
```

### 登録フロー チェックリスト
```
□ 商談DB のステータスを「成約」に更新
□ 人脈DB → クライアントDB 昇格
□ 契約書テンプレート選択・生成
□ 契約書 PDF 格納
□ 電子署名の送付
□ クライアントフォルダ作成
□ CFO 通知（初回請求）
□ Success Agent 通知（オンボーディング）
```

## 契約更新管理（renew モード）

### 更新アラートタイムライン

| タイミング | アクション |
|---|---|
| 更新90日前 | 更新検討フラグ設定、ヘルススコア確認 |
| 更新60日前 | 🟡 アラート発動、更新提案の準備開始 |
| 更新45日前 | 更新提案書の作成、プラン変更シミュレーション |
| 更新30日前 | 🔴 クライアントへ更新案内、面談設定 |
| 更新14日前 | 最終確認、未回答の場合は電話フォロー |
| 更新7日前 | 緊急対応 — CEO直接連絡 |
| 更新当日 | 契約書更新 or 終了処理 |

### 更新判定ロジック
```
days_to_renewal = contract_end_date - today

renewal_action = case:
    days_to_renewal > 90:   "通常管理"
    days_to_renewal > 60:   "更新準備開始"
    days_to_renewal > 30:   "更新提案・面談"
    days_to_renewal > 14:   "最終確認"
    days_to_renewal > 7:    "緊急フォロー"
    days_to_renewal <= 7:   "CEO介入"
```

### プラン変更シミュレーション
```
更新時に以下のシナリオを提示:
1. 現プラン継続: ¥{current_amount}/月 × {contract_months}ヶ月
2. プランアップグレード: ¥{upgrade_amount}/月（差額: +¥{diff}）
3. 年間契約割引: ¥{annual_amount}/月（{discount}%OFF）
4. 追加サービス: +¥{additional_amount}/月
```

### 更新提案書の自動生成
```
提案内容:
- 契約期間中の成果サマリー（Success Agent から取得）
- 次期の提案プラン・金額
- プラン変更のメリット
- 年間契約の割引特典
```

## 契約終了管理（terminate モード）

### 解約プロセス

#### フェーズ1: 解約申し出の受理
```
1. 解約理由のヒアリング
   | 理由カテゴリ | 詳細 |
   |------------|------|
   | コスト | 予算削減、費用対効果 |
   | 品質 | サービス不満足、期待値不一致 |
   | ニーズ変化 | 社内体制変更、事業方針転換 |
   | 競合 | 他社サービスへの乗り換え |
   | その他 | 合併、事業停止等 |
2. 解約理由を Intelligence Agent に記録
```

#### フェーズ2: リテンション施策（該当する場合）
```
ヘルスが40以上の場合、リテンション施策を検討:
- プラン変更（ダウングレード提案）
- 一時休止（最大3ヶ月）
- 追加サポート提供
- CEO面談の設定
```

#### フェーズ3: 引き継ぎ
```
□ 最終成果物の納品確認
□ データ・資料の引き渡し
□ アカウント・アクセス権の整理
□ 最終請求の確認（CFO連携）
□ 退出インタビュー実施
□ クライアントDBのステータスを「解約」に更新
```

#### フェーズ4: Win-back 設計
```
解約後の Win-back スケジュール:
| タイミング | アクション |
|---|---|
| 解約後30日 | お礼・フィードバック依頼メール |
| 解約後90日 | 新サービス・改善報告の共有 |
| 解約後180日 | 再提案（新プラン・特別条件） |
| 解約後365日 | 年次レビューメール |
```

## コンプライアンスチェック（compliance モード）

### チェック項目

| チェック項目 | 判定条件 | ステータス |
|---|---|---|
| 契約書格納 | `{{WITHAI_ROOT}}/clients/{会社名}/contracts/` にPDFが存在 | ✅/❌ |
| 署名確認 | 署名済み契約書が格納されている | ✅/❌ |
| 契約期間の整合性 | クライアントDBの期間と契約書の期間が一致 | ✅/❌ |
| NDA期限 | NDAの有効期限が切れていない | ✅/❌ |
| 金額の整合性 | 契約書の金額とクライアントDBの金額が一致 | ✅/❌ |

### コンプライアンスレポート
```
compliance_score = passed_checks / total_checks * 100
```

| スコア | ステータス |
|---|---|
| 100% | ✅ 完全準拠 |
| 80-99% | 🟡 軽微な不備あり |
| 80%未満 | 🔴 要対応 |

## 実行モード一覧

| モード | コマンド | 動作 |
|---|---|---|
| 契約登録 | `register [会社名]` | 新規契約の登録フロー実行 |
| 契約更新 | `renew [会社名]` | 更新シナリオ提示・更新フロー |
| 契約終了 | `terminate [会社名]` | 解約プロセス実行 |
| コンプライアンス | `compliance` | 全契約のコンプライアンスチェック |
| 契約一覧 | `list` | 全契約の一覧表示 |

### 処理フロー

1. 環境変数を読み込む
2. クライアントDB を取得
3. 実行モードに応じて:
   - register: 商談DB確認 → 昇格 → 契約書生成 → 後処理
   - renew: 更新対象の確認 → シミュレーション → 提案準備
   - terminate: 理由ヒアリング → 引き継ぎ → Win-back設計
   - compliance: ローカルファイル + DB の整合性チェック
   - list: 全契約を一覧表示
4. 必要に応じて配下スキルを呼び出し
