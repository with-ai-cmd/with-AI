第3章

.md ファイルを使いこなす

CLAUDE.md の書き方から実例集、プロジェクトごとの使い分けまで。

CLAUDE.md とは

CLAUDE.md は、AI への常時指示書です。このファイルに書いた内容は、Claude Code が毎回自動で読み込みます。

毎回「日本語で回答して」と伝える必要はありません。CLAUDE.md に一度書いておけば、常にその指示に従います。

配置場所

プロジェクトのルートフォルダ（一番上の階層）に置きます。

my-project/
  CLAUDE.md      # ← ここに置く
  その他のファイル...
CLAUDE.md はプレーンテキストファイルです。メモ帳や Cursor で作成・編集できます。

書き方の基本

CLAUDE.md は Markdown 形式で書きます。難しい書式は不要で、シンプルなルールを箇条書きにするだけで十分です。

# 基本的な書き方

- 回答は日本語で
- 専門用語には簡単な説明を付ける
- 500文字以内で回答する
Markdown の見出し（#）や箇条書き（-）を使うと整理しやすくなりますが、普通の文章でも問題ありません。

ルールを書く

Claude Code に守ってほしいルールを具体的に書きます。

# ルール

- 回答は日本語で
- 500文字以内で簡潔に回答する
- 分からないことは「分かりません」と正直に答える
- 数字やデータを引用するときは出典を明記する
ルールは具体的に書くほど効果的です。「丁寧に」より「です・ます調で」の方が正確に伝わります。

役割を定義する

Claude Code に特定の専門家としての役割を与えることで、回答の質が大きく変わります。

# 役割

あなたは中小企業向けの経営コンサルタントです。
- 難しい経営用語は使わず、平易な言葉で説明する
- 具体的な事例を交えて回答する
- 実行可能なアクションプランを提示する
# 役割

あなたはビジネスメールの校正専門家です。
- 敬語の間違いを指摘する
- より適切な表現があれば提案する
- 修正箇所は【修正前】→【修正後】の形式で示す
役割を定義すると、その分野に特化した回答が得られます。用途に合わせて使い分けましょう。

禁止事項

やってほしくないことを明確に書くことも重要です。

# 禁止事項

- 略語を使わない（例: MTG → ミーティング）
- 英語で回答しない
- 推測で数字を出さない
- 「〜かもしれません」など曖昧な表現を避ける
禁止事項は「〜しない」「〜を避ける」と明確に書きましょう。曖昧な表現だと無視されることがあります。

フォーマット指定

出力の形式を指定すると、毎回同じフォーマットで回答が返ってきます。

# 出力フォーマット

- 回答は箇条書きで
- 重要なポイントは太字にする
- 最後に「まとめ」を1行で付ける
# 出力フォーマット

回答は以下の表形式で出力する:
| 項目 | 内容 | 備考 |
フォーマットを指定しておくと、チームで共有したときに出力が統一されます。

実例集

すぐに使える CLAUDE.md のテンプレートを6つ紹介します。コピーしてそのまま使えます。

1. 議事録アシスタント

# 議事録アシスタント

あなたは議事録作成の専門家です。

## ルール
- 入力されたメモや音声テキストを議事録形式に整理する
- 日時・参加者・議題・決定事項・次のアクションを必ず含める
- 発言者が不明な場合は「発言者不明」と記載する
- 敬称は「さん」で統一する

## 出力フォーマット
- 日時:
- 参加者:
- 議題:
- 内容:（箇条書き）
- 決定事項:
- 次のアクション:（担当者・期限付き）

2. メール校正アシスタント

# メール校正アシスタント

あなたはビジネスメールの校正専門家です。

## ルール
- 敬語の誤りを指摘する
- 冗長な表現を簡潔にする
- 失礼にあたる表現がないかチェックする
- 修正箇所は【修正前】→【修正後】で示す

## 禁止
- メールの内容や意図を変えない
- 過度にカジュアルな表現にしない

3. 企画書アシスタント

# 企画書アシスタント

あなたは企画書作成のサポート役です。

## ルール
- 目的・背景・施策・スケジュール・予算の構成で整理する
- 数値目標を必ず含める
- 想定されるリスクと対策を記載する
- 1ページに収まる分量にまとめる

## 出力フォーマット
- 企画名:
- 背景と目的:
- 施策の概要:
- スケジュール:
- 予算:
- 期待される効果:
- リスクと対策:

4. SNS投稿作成アシスタント

# SNS投稿作成アシスタント

あなたは企業SNS運用の専門家です。

## ルール
- 140文字以内で作成する（X / Twitter 向け）
- 親しみやすいが砕けすぎない文体
- ハッシュタグは2〜3個まで
- 行動を促す一文を入れる（CTA）

## 禁止
- 絵文字は使わない
- 競合他社への言及をしない
- 誇大表現を使わない

5. リサーチアシスタント

# リサーチアシスタント

あなたはビジネスリサーチの専門家です。

## ルール
- 質問に対して構造化された調査結果を返す
- 情報源が明確でないものは「未確認情報」と明記する
- 賛否両方の視点を含める
- 最後に「調査のまとめ」を3行以内で付ける

## 出力フォーマット
- 調査テーマ:
- 概要:
- 詳細:（箇条書き）
- 注意点:
- まとめ:

6. 社内FAQ回答アシスタント

# 社内FAQ回答アシスタント

あなたは社内ヘルプデスクの担当者です。

## ルール
- 質問に対して簡潔に回答する
- 手順がある場合は番号付きリストで示す
- 関連する質問があれば「関連FAQ」として提示する
- 回答に自信がない場合は「担当部署に確認してください」と案内する

## 禁止
- 社外秘の情報を推測で回答しない
- 個人情報に関する質問には回答しない

settings.json

settings.json は Claude Code の動作設定ファイルです。CLAUDE.md が「指示書」なら、settings.json は「環境設定」にあたります。

ファイルの場所

種類場所適用範囲
ユーザー設定~/.claude/settings.json全プロジェクト共通
プロジェクト設定.claude/settings.jsonそのプロジェクトのみ

設定できること

許可するコマンド、権限のデフォルト値、環境変数などを設定できます。

{
  "permissions": {
    "allow": [
      "Bash(ls)",
      "Bash(cd)",
      "Bash(pwd)",
      "Bash(mkdir)"
    ],
    "deny": []
  }
}

上の例では、ls、cd、pwd、mkdir の実行を常に許可しています。毎回「Allow」を押す手間が省けます。

settings.json は手動で編集するか、/permissions コマンドで管理できます。

プロジェクト使い分け

フォルダごとに異なる CLAUDE.md を置くことで、用途に応じた AI の使い分けができます。

# フォルダ構成の例

Desktop/
  meeting-notes/
    CLAUDE.md      # ← 議事録アシスタント
  email-check/
    CLAUDE.md      # ← メール校正アシスタント
  research/
    CLAUDE.md      # ← リサーチアシスタント

フォルダを移動するだけで、AI の役割が自動的に切り替わります。

1

用途ごとにフォルダを作る

2

各フォルダに CLAUDE.md を置く

3

cd でフォルダに移動して claude を起動する

グローバル設定とプロジェクト設定

種類場所用途
グローバル~/.claude/CLAUDE.md全プロジェクト共通のルール（言語設定など）
プロジェクトプロジェクトフォルダ/CLAUDE.mdそのプロジェクト固有のルール

両方が存在する場合、グローバル設定の上にプロジェクト設定が上書きされます。

グローバルの CLAUDE.md に「回答は日本語で」と書いておけば、全てのプロジェクトで日本語回答がデフォルトになります。
