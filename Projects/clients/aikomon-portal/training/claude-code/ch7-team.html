チーム運用・応用

チーム全体で Claude Code を安全・効率的に活用するための運用ガイドです。

チームルール設計

チームで Claude Code を使う場合、ルールを決めておくことが重要です。

なぜルールが必要なのか

品質の統一

メンバーごとに出力の質がバラつくのを防ぐ

セキュリティ

機密情報の取り扱いミスを防止する

効率化

毎回同じ設定をする手間を省く

知識の共有

うまくいったプロンプトやルールをチーム全体で活用

ルールで決めるべきこと

項目内容の例
コーディングスタイル言語、インデント、命名規則
コミットメッセージフォーマット、言語（日本語/英語）
禁止操作本番環境への直接デプロイ、force push
機密情報API キーやパスワードの取り扱い
レビュー基準AI 生成コードのレビュー必須かどうか

CLAUDE.md の共有

チームルールは CLAUDE.md に書いて、Git で共有するのが最も効率的です。

設定ファイルの使い分け

ファイル範囲Git に含める
CLAUDE.mdプロジェクト全体のルール含める
.claude/settings.jsonプロジェクト共通設定含める
.claude/settings.local.json個人設定（ローカル専用）含めない
~/.claude/settings.jsonユーザー全体設定含めない
settings.json はチーム共通のルール、settings.local.json は個人のカスタマイズに使います。

CLAUDE.md の例

# プロジェクトルール

## コーディング規約
- TypeScript を使用すること
- 変数名は camelCase
- コメントは日本語で記述

## Git ルール
- コミットメッセージは日本語
- main ブランチへの直接 push 禁止
- PR にはレビュー必須

## セキュリティ
- API キーをコードに書かない
- .env ファイルは Git にコミットしない

カスタムコマンド

よく使う指示をコマンドとして登録しておくと、チーム全員が同じ操作を簡単に実行できます。

コマンドの保存場所

カスタムコマンドは .claude/commands/ フォルダ内に Markdown ファイルとして保存します。

# フォルダ構成
.claude/
  commands/
    review.md      # /review コマンド
    memo.md        # /memo コマンド
    mail.md        # /mail コマンド

例1: /review（コードレビュー）

# .claude/commands/review.md

変更されたファイルをレビューしてください。
以下の観点でチェックし、問題があれば指摘してください。

1. バグやエラーの可能性
2. セキュリティ上の問題
3. パフォーマンスの改善点
4. コーディング規約への準拠

問題がなければ「問題なし」と報告してください。

例2: /memo（議事録作成）

# .claude/commands/memo.md

以下の形式で議事録を作成してください。

## 会議情報
- 日時:
- 参加者:
- 議題:

## 決定事項

## TODO
- [ ] 担当者: タスク内容（期限）

## 次回予定

例3: /mail（メール下書き）

# .claude/commands/mail.md

以下の条件でビジネスメールの下書きを作成してください。

- 敬語を使う
- 簡潔に要点をまとめる
- 件名も提案する
- 署名は省略する

$ARGUMENTS の内容をもとにメールを作成してください。
$ARGUMENTS はコマンド実行時に追加したテキストが入ります。例: /mail 納品日の変更について連絡

セキュリティ設定

settings.json を使って、Claude Code の動作に制限をかけることができます。

コマンド制限

実行を許可するコマンドと禁止するコマンドを設定できます。

// .claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(git status)",
      "Bash(git diff)",
      "Bash(npm test)",
      "Bash(npm run build)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force)",
      "Bash(curl*)"
    ]
  }
}

ファイルアクセス制限

Claude Code がアクセスできるファイルやフォルダを制限することも可能です。

本番環境の設定ファイルや、認証情報を含むファイルへのアクセスは制限しておきましょう。

エンタープライズ向けチェックリスト

  • settings.json でコマンドの許可・禁止リストを設定している
  • 本番環境への直接操作を禁止している
  • 機密ファイルへのアクセスを制限している
  • API キーの管理ルールをチームに周知している
  • AI 生成コードのレビュープロセスを定めている
  • インシデント発生時の対応フローを用意している

運用ベストプラクティス

日々の運用で意識すべきポイントをまとめます。

フォルダ構成を整える

# 推奨フォルダ構成
project/
  CLAUDE.md              # チームルール
  .claude/
    settings.json        # 共通設定
    settings.local.json  # 個人設定（Git 対象外）
    commands/
      review.md          # カスタムコマンド
      memo.md
  .mcp.json              # MCP サーバー設定
  .env                   # API キー（Git 対象外）
  .gitignore             # .env を除外

CLAUDE.md は少しずつ育てる

1

最初は基本ルールだけ書く（5〜10行で十分）

2

日々の業務で「毎回同じことを言っている」と気づいたら追記

3

チームメンバーからのフィードバックを反映

4

定期的に見直し、古いルールを整理する

完璧な CLAUDE.md を最初から作ろうとしないでください。実際に使いながら改善していくのが最も効果的です。

/compact でメンテナンスする

長い会話が続くと Claude Code の応答が遅くなったり、文脈を忘れたりすることがあります。

# 会話が長くなったら実行
/compact
/compact は会話の要点を圧縮して、パフォーマンスを改善します。作業の区切りで実行するのがおすすめです。
